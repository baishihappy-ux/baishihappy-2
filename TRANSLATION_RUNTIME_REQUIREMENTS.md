# 翻译运行时需求存档

本文件存档当前翻译运行时、缓存、实时补翻、未读提示、通知、轻量化和真实测试要求。

这些条目是后续落代码的约束，不是已全部完成的功能。实现时必须同时考虑用户体验、CPU 占用、运行内存占用和多开稳定性。

## 1. 气泡识别

只按位置判断是不是聊天气泡，不按英文词过滤。气泡里的非中文内容进入缓存优先翻译系统。

## 2. 缓存秒显

重启、切换多开、切换会话、滚动历史时，批量读缓存、批量注入，目标是接近友商那种“往回拉几屏也秒出”。

缓存命中不调用翻译接口。

## 3. 实时补翻队列

缓存没有的内容才进入实时翻译队列。

队列必须支持优先级、去重、并发限制和失败重试，避免多个多开同时触发大量请求导致卡顿或 CPU 占用升高。

## 4. 已方发出英文气泡回翻

已方发送出去的英文气泡必须调用英译中接口回翻，不能直接使用用户刚输入的原中文。

目的：让用户看到英文真实表达回译后的中文，帮助下次输入更符合英文表达习惯的中文。

## 5. 中文输入回车体验

中文输入回车后先中译英并写入输入框，再次回车发送。

Signal、WhatsApp、Telegram 的用户体验要尽量一致。

## 6. 刷新按钮语义

刷新按钮必须绕过缓存，强制重新调用翻译接口，并替换旧译文。

刷新不能被旧翻译节点挡住，也不能继续显示旧的英文译英文坏结果。

## 7. 坏缓存清理

纯英文译纯英文、不合格译文不能显示，不能继续写入缓存。

磁盘里已有的旧坏缓存需要后台清理，不应在前台操作时阻塞用户。

## 8. 会话识别

缓存必须按多开账号和会话对象隔离。

联系人标题、号码、备注、平台会话 ID 要尽量稳定，避免不同客户串缓存。

## 9. 滚动历史批量策略

滚动历史时先批量查缓存并注入页面。

缓存没有的历史气泡再进入低优先级后台补翻队列，不能让用户一条条等待。

## 10. 三个平台统一体验

WhatsApp、Telegram、Signal 都要具备同一套核心体验：

- 缓存秒显
- 实时补翻
- 中文回车转英文
- 已方英文气泡回翻
- 刷新重译
- 未读提示

## 11. 未读提示

未读提示以 WhatsApp、Telegram、Signal 平台自己的未读/已读状态为事实源。

程序只同步并展示平台状态，不靠自己猜。

左侧三个大 logo：平台下任意多开有未读就闪烁。

顶部多开标签：对应多开卡片左上角显示红底白字未读数量，位置不能和关闭、修改按钮冲突。

## 12. 通知可靠性

已打开的多开必须尽量保持接收通知。

通知点击不能再触发打开新的 Signal 实例这类副作用。

未读提示不依赖系统通知点击，而是依赖平台原生未读/已读状态同步。

## 13. 轻量化

程序必须尽量少吃 CPU、少吃运行内存。

要求：

- 当前激活实例高优先级。
- 后台实例低频检测。
- 不全量扫描历史 DOM。
- 不让全部缓存常驻内存。
- 磁盘保存全量缓存，内存只保留当前会话、最近会话和必要索引。
- 历史补翻、坏缓存清理、企业资产写入放后台空闲批处理。
- UI 闪烁用 CSS 动画，不用 JS 高频定时器。

## 14. 企业聊天资产

后台静默沉淀企业资产：聊天对象、方向、时间、原文、译文、引用、媒体索引等。

前端不要出现企业聊天资产相关设置、按钮或文案。

## 15. 关闭多开无残留进程

关闭 Signal、WhatsApp、Telegram 多开后，相关运行资源不能残留。

关闭运行资源不等于删除聊天记录、翻译缓存或企业资产。

## 16. Signal 显示稳定性

Signal 最小化恢复、切换多开、切换平台后要稳定贴回主程序工作区。

不能闪烁、不能消失、不能在恢复或切换时拉爆 CPU。

## 17. 真实 UI 测试

不能只用构建检查代替真实验证。

必须实际测试：

- 重启主程序
- 切换多开
- 切换会话
- 向上滚动历史
- 新发中文
- 收到对方回复
- 点击刷新按钮
- 未读数增减
- 左侧 logo 闪烁
- 顶部多开标签未读徽标
- 关闭多开后进程无残留
- Signal 恢复和切换显示稳定

只有真实 UI 路径测过后，才能说对应体验已验证。

## 具体实现参数

以下参数是后续落代码的初始默认值。真实 UI 测试发现卡顿、漏翻、误翻时，只能在这些边界内有依据地调整，不能随手改成无上限扫描或无上限请求。

### 1. 气泡识别

需求：只翻译聊天气泡里的非中文内容，不按英文词过滤。

做法：

- 每个平台实现一个 `messageAdapter`（消息适配器）。
- 适配器只输出已确认的气泡候选，不输出聊天列表、顶部栏、输入框、菜单、联系人卡片。
- 候选统一字段：`profileId`、`platform`、`contactId`、`contactTitle`、`messageKey`、`sourceHash`、`sourceText`、`direction`、`messagePart`、`timestamp`、`mountSelectorHint`。

参数：

- 文本最短长度：2 个字符。
- 文本最长长度：4000 个字符。
- 扫描范围：当前可视区，加上上方 2 屏、下方 1 屏作为预取范围。
- 禁止使用 `voice`、`message`、`online`、`search` 这类英文词过滤气泡内容。

验收：

- `Talk to you later`、`Just checking in`、`See you soon` 这类短句不能被过滤掉。
- 输入框提示和聊天列表不能被当成气泡翻译。

### 2. 缓存秒显

需求：重启、切换多开、切换会话、滚动历史时先批量显示缓存译文。

做法：

- 磁盘保存全量缓存。
- 内存只保留热缓存：当前会话、最近会话、必要索引。
- 页面注入先查缓存，缓存命中直接显示，不进实时翻译队列。
- 磁盘缓存必须按 `platform/profileId/contactKey/chunk`（平台/多开/会话/分块）切块，不能让所有多开读写同一个大缓存文件。
- 每个会话要保存手动翻阅进度，用户翻到哪里，就把该会话历史缓存进度保存到哪里。
- 翻译热缓存只负责秒显和避免重复翻译；企业资产归档必须另走 `EnterpriseChatAssets`（企业聊天资产路径），不能混进 `TranslationCache`（翻译缓存路径）。

参数：

- 当前会话预加载：最近 1000 条缓存。
- 最近会话保留：全部保留索引；热数据按最近使用顺序进入内存。
- 最近会话热加载量：每个会话最近 1000 条缓存。
- 全局内存热缓存上限：120000 条译文记录或 320 MB，先到为准。
- 磁盘分块大小：每块约 1000 条记录或 1-2 MB，先到为准。
- 禁止把 320 MB 作为单个缓存文件大小；320 MB 只代表全局内存热缓存上限。
- 单个多开的指纹浏览器渲染热缓存：最多 1500 条或 15 MB，先到为准。这个缓存只存在当前指纹浏览器进程内存中，只服务会话内切换后的快速重渲染，不得把译文明文写入 `localStorage` 或其它网页持久存储。
- 指纹浏览器渲染缓存不是权威数据源；`TranslationCache`（翻译缓存路径）仍然是权威缓存。
- 缓存注入单批：最多 200 个候选节点。
- 单次触发最多处理：800 个候选节点。
- 近视窗范围：上方 2 屏、下方 1 屏。这个范围只用于缓存命中显示，不扩大实时翻译接口扫描范围。
- `MutationObserver`（页面变化监听器）调度：进入下一帧处理；如果正在运行，只合并成一次待处理请求。
- 滚动触发节流：180 ms。

验收：

- 切换多开后当前屏缓存译文应立即出现。
- 向上滚动几屏时，命中缓存的译文不应一条条慢慢出现。

### 3. 实时补翻队列

需求：缓存没有的内容才调用翻译接口。

做法：

- 建立统一 `translationQueue`（翻译队列）。
- 所有平台共用全局并发限制。
- 同一 `profileId + sourceHash` 只允许一个进行中的翻译任务。

参数：

- 全局翻译并发：2。
- 单个多开并发：1。
- 当前可视区缓存未命中任务：每轮最多 8 条。
- 历史后台补翻：每个空闲批次最多 3 条。
- 失败重试：最多 2 次。
- 重试间隔：首次 5 秒，第二次 30 秒。
- 同一原文去重窗口：10 分钟。

优先级：

1. 手动刷新。
2. 中文回车转英文。
3. 已方刚发送英文气泡回翻。
4. 对方新增回复英译中。
5. 首次历史消息低优先级补翻。

验收：

- 多开同时有消息时，CPU 不应持续拉满。
- 手动刷新不能排在大量历史补翻后面。

### 4. 已方发出英文气泡回翻

需求：已方发出的英文气泡也必须调用英译中接口。

做法：

- 扫描到 `direction = outgoing` 的英文气泡后，照常进入缓存优先逻辑。
- 缓存没有时调用英译中接口。
- 禁止把用户刚输入的中文原文直接作为该气泡中文译文。

参数：

- 已方刚发送气泡优先级高于历史补翻。
- 发送后 10 秒内扫描到的 outgoing 气泡标记为“刚发送”，优先补翻。

验收：

- 用户发送英文后，气泡下方显示的是英文回翻结果，不是原中文。

### 5. 中文输入回车体验

需求：中文输入回车后先中译英写入输入框，再次回车发送。

做法：

- 中文检测只在输入框焦点内启用。
- 翻译完成后替换输入框内容为英文。
- 第二次回车放行给平台原生发送。

参数：

- 中文输入预翻译防抖：300 ms。
- Signal 输入框活跃检测：仅聚焦或当前激活 Signal 多开时运行。
- Signal 输入检测轮询：活跃时 360 ms；非活跃时不做输入框轮询。
- 翻译超时提示：15 秒。

验收：

- Signal、WhatsApp、Telegram 都要支持中文回车转英文。
- 没有中文时回车不能被拦截。

### 6. 刷新按钮语义

需求：刷新按钮强制重新翻译并替换旧译文。

做法：

- 刷新按钮点击后绕过缓存。
- 翻译成功后删除旧 `.df-chat-translation` 节点并写入新节点。
- 新结果通过质量检查后写入缓存，覆盖旧缓存。

参数：

- 手动刷新优先级最高。
- 每次最多处理 3 个手动刷新候选，避免一个气泡内引用和正文造成过多并发。
- 刷新按钮视觉反馈保持 1600 ms。

验收：

- 英文译英文坏结果点击刷新后不能继续显示旧结果。

### 7. 坏缓存清理

需求：坏缓存不能显示，不能继续污染磁盘。

做法：

- 读取缓存时做质量检查。
- 发现坏缓存先从内存剔除。
- 磁盘清理放后台空闲任务。

参数：

- 坏缓存判断：源文本有英文，译文没有中文；或译文与源文忽略大小写完全相同。
- 后台清理间隔：10 分钟。
- 每次最多处理 500 条缓存记录。
- 用户正在滚动、输入、切换多开时暂停清理。

验收：

- `Meeting is over` 缓存成 `Meeting is over` 这种记录不能继续显示。

### 8. 会话识别

需求：缓存按多开账号和会话对象隔离，不能串客户。

做法：

- 缓存主键包含：`platform + profileId + contactKey + sourceHash`。
- `contactKey` 优先级：平台会话 ID > 电话号 > username > 标题哈希。
- 标题只能作为兜底，不能长期作为唯一可靠来源。

参数：

- 联系人标题采样最多读取 6 个候选节点。
- 电话号提取最少 7 位数字。
- contactKey 变化时保留旧缓存，但新写入走新 key。

验收：

- 同一个英文句子出现在不同客户会话里，缓存不能串到错误客户名下。

### 9. 滚动历史批量策略

需求：往回拉几屏也能尽快显示缓存译文。

做法：

- 滚动触发缓存批量注入。
- 历史渲染预取必须按批次加载译文：每批最多 100 个会话气泡，命中缓存后一次性提交显示，不能视觉上几条几条断续出现。
- 缓存没有的历史气泡进入低优先级补翻队列。
- 不因为历史补翻阻塞当前可见新消息。
- 会话内手动翻阅到的位置必须持久化，下次回到该会话时从已翻阅范围附近恢复缓存加载。

参数：

- 滚动节流：180 ms。
- 上方预取：2 屏。
- 下方预取：1 屏。
- 历史渲染预取扫描范围：上方 3 屏、下方 2 屏。
- 历史渲染预取批次：最多 100 个气泡译文。
- 每次滚动缓存注入上限：800 个候选节点。
- 历史补翻每轮最多 3 条。
- 手动翻阅进度字段：`oldestViewedKey`、`oldestViewedAt`、`lastViewedRange`。

验收：

- 快速向上滚动时，已缓存译文应批量出现，不能每条等待接口。

### 10. 三个平台统一体验

需求：WhatsApp、Telegram、Signal 的翻译体验一致。

做法：

- 使用统一队列、统一缓存、统一质量检查、统一刷新语义。
- 平台差异只放在 adapter 内。

参数：

- 平台 adapter 输出同一候选结构。
- 平台差异不能泄漏到缓存键规则之外。

验收：

- 三个平台都要完成缓存秒显、实时补翻、中文回车、已方回翻、刷新、未读提示。

### 11. 未读提示

需求：未读数以平台原生未读/已读状态为事实源。

做法：

- 每个平台实现 `unreadAdapter`（未读适配器）。
- 读取平台自己的未读源：标题、聊天列表未读徽标、平台 DOM 状态或 Signal 独立窗口状态。
- 维护 `unreadByProfileId` 和 `unreadByApp`。

参数：

- 当前激活多开未读同步：2 秒。
- 后台多开未读同步：10 秒。
- 主程序最小化时未读同步：15 秒。
- 状态更新防抖：300 ms。
- 顶部标签徽标：1-99 显示数字，超过 99 显示 `99+`。
- 徽标样式：红底白字，放在多开图标卡片左上角，不能遮挡关闭和修改按钮。
- 左侧 logo 闪烁周期：1.2 秒 CSS 动画。

验收：

- 某个平台任意多开有未读时，左侧对应 logo 闪烁。
- 进入平台后，顶部对应多开显示各自未读数。
- 平台已读状态清零后，我们的 UI 也清零。

### 12. 通知可靠性

需求：已打开多开要尽量保持接收通知，未读提示不依赖系统通知点击。

做法：

- 不通过系统通知点击来定位实例。
- Signal 通知点击不再用于激活实例，避免打开新 Signal。
- 未读状态由平台原生未读源同步。

参数：

- 已打开多开不得因省资源被自动关闭。
- 后台多开检测低频运行，但保留平台接收通知能力。

验收：

- 打开的实例能收到新消息提示。
- 点击系统通知不会创建新的 Signal 实例。

### 13. 轻量化

需求：在不影响体验和通知的前提下，尽量少吃 CPU 和内存。

做法：

- 当前激活实例高优先级，后台实例低频。
- DOM 扫描只扫候选范围，不全量扫历史。
- 缓存磁盘全量保存，内存只保留热数据。
- 企业资产写入、坏缓存清理、历史补翻都走后台批处理。

参数：

- 全局内存热缓存上限：120000 条或 320 MB。
- 后台实例扫描间隔不低于 10 秒。
- 当前实例扫描间隔不低于 2 秒，除非是用户手动刷新或输入事件。
- 企业资产写入：15 秒或 200 条触发一次刷盘。
- 任何后台任务单次主线程连续运行时间目标低于 16 ms。

验收：

- 切换多开、恢复主程序、滚动历史时不能持续 CPU 100%。
- 空闲时 CPU 应接近静默状态。

### 14. 企业聊天资产

需求：后台静默沉淀聊天资产，不影响前台。

做法：

- 翻译缓存写入后，异步排队写企业资产。
- 资产记录包含联系人、方向、时间、原文、译文、引用、媒体索引。
- 前端不显示企业资产设置、按钮或文案。
- 企业资产归档必须另走 `EnterpriseChatAssets`（企业聊天资产路径），不和翻译热缓存混在一起。
- `TranslationCache`（翻译缓存路径）只服务前台秒显和去重；`EnterpriseChatAssets` 只服务企业资产沉淀、后续检索、复盘和 AI 数据准备。

参数：

- 写入批次：最多 200 条。
- 延迟刷盘：15 秒。
- 文件格式当前用 `JSONL`，后续可迁移到 SQLite。

验收：

- 前台 UI 没有企业资产入口。
- 关闭多开不会删除已沉淀资产。

### 15. 关闭多开无残留进程

需求：关闭多开释放运行资源，但保留数据资产。

做法：

- WhatsApp/Telegram 关闭后销毁对应 webview、定时器、翻译任务和缓存热数据。
- Signal 关闭后按 profileId 精准关闭对应 Signal 子进程。

参数：

- 关闭后 2 秒检查是否仍有对应资源。
- Signal 正常关闭失败后 5 秒执行强制终止。
- 不删除磁盘缓存、聊天资产、登录态，除非用户明确要求清理数据。

验收：

- 关闭多开后任务管理器里没有对应残留运行实例。

### 16. Signal 显示稳定性

需求：Signal 切换和恢复不闪、不消失、不拉爆 CPU。

做法：

- Signal 独立窗口只在当前 Signal profile 激活时贴回工作区。
- 隐藏和显示使用已有窗口控制通道。
- 切换时避免短时间多次 show/hide/bounds。

参数：

- bounds 同步防抖：80 ms，保持当前代码参数。
- 恢复主程序后 Signal 同步最多连续尝试 3 次。
- 恢复/切换 burst：100 ms，保持当前代码参数。
- 不再使用 220 ms show/hide 最小间隔作为规划参数。
- 稳态同步不要用 800 ms 永久轮询；只在窗口变化、切换、恢复时触发短 burst。

验收：

- 主程序最小化再恢复，当前 Signal 仍在工作区。
- SG01/SG02 切换不消失、不闪烁、不 CPU 爆表。

### 18. 语音条英文转中文（后续版本）

状态：后续版本规划，当前版本不实现、不验收。

需求：对方发送英文语音条时，在不改变现有文字气泡体验的前提下，把语音内容转成中文显示在对应语音气泡下方。

目标体验：

- 用户看到英文语音条后，程序识别语音内容。
- 先执行语音转文字，再执行英文转中文。
- 中文译文显示在该语音条气泡下方，视觉风格与文字气泡译文一致。
- 译文写入翻译缓存；重启、切换多开、翻历史时优先缓存秒显，不重复消耗接口。
- 企业聊天资产后续可记录语音附件索引、英文转写文本和中文译文，但不把音频二进制直接塞进消息文本记录。

建议实施顺序：

- 第一阶段：Telegram。先用 Telegram Web 语音条 DOM 跑通定位、音频提取、转写、翻译、缓存和注入。
- 第二阶段：WhatsApp。复用同一套缓存和气泡注入策略，补 WhatsApp 语音条定位和音频资源提取。
- 第三阶段：Signal。等前两套稳定后再做，因为 Signal 是独立窗口，语音资源和注入控制更复杂。

技术链路：

- 语音条定位。
- 音频文件或音频资源提取。
- STT（语音转文字）得到英文文本。
- 英文转中文。
- 写入 `TranslationCache`（翻译缓存路径）。
- 异步写入 `EnterpriseChatAssets`（企业聊天资产路径）。
- 把中文译文挂到对应语音条气泡下方。

原则：

- 当前客户验收版先不做语音条翻译。
- 不为了语音条功能影响文字气泡翻译、未读预览、输入中文回车翻译和 Signal 显示稳定性。
- 语音转文字接口需要单独评估成本、速度、隐私和准确率；本地模型会占硬件，云端 STT 会增加接口成本和隐私边界。
- 缓存键必须绑定平台、profileId、会话对象、语音消息唯一标识或音频内容哈希，避免重复转写和重复翻译。
- 未完成真实 UI 手动测试前，只能标记为“已规划”，不能标记为“已支持”。
