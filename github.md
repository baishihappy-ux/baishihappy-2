# GitHub 通版与灾难恢复规则

英文执行参考见：`github.en.md`。

## 两条最高原则

以下两条是所有 GitHub 同步、提交和推送工作的最高判断标准：

1. **本机电脑遗失后可恢复**
   GitHub 必须保存足够完整的项目源码和开发资料。换到另一台电脑后，应能重新安装依赖、构建、测试并继续开发，不能只是保存一份无法工作的代码片段。
2. **GitHub 只保存 `maoyi` 通版**
   通版与当前业务工作区保持相同功能、布局、交互、架构和测试能力，只去除业务版企业名称、标识及企业专属展示，统一使用 `maoyi`。

其他规则都是这两条原则下的执行边界，不得用“安全清理”作为遗漏功能源码的理由，也不得用“完整恢复”作为上传密钥或客户数据的理由。

## 通版定义

通版不是精简版、演示版或删功能版。

通版必须同步业务工作区的完整产品能力，包括但不限于：

- Signal / WhatsApp / Telegram 多开。
- 输入翻译、气泡翻译、缓存、刷新和未读预览。
- 主题、工作区、窗口控制和锁屏。
- Signal 显示、隐藏、贴合、进程和多开管理逻辑。
- 授权、机器绑定、安全校验及其开发工具源码。
- 构建、测试、打包工具和项目自有 Codex 技能副本。
- 中英文长期文档和关键产品设计记录。

通版允许存在的差异仅限于：

- 品牌名称统一为 `maoyi`。
- 企业 logo、ico、主题图和品牌图片替换为通版资源。
- 企业专属显示文案、应用名和数据目录名替换为通版名称。

密钥注入和客户交付产物不属于“业务版与通版的功能差异”。它们是任何版本都不得提交到 GitHub 的本地安全数据。

## 恢复范围

GitHub 通版必须包含：

- 完整源码，包括主进程、渲染进程、授权端、Signal 改造和平台注入逻辑。
- 不含真实秘密值的配置模板和环境变量示例。
- 依赖清单和锁文件。
- 构建、测试、打包、白标同步和安全检查脚本。
- 数据结构、缓存、授权、套装和升级流程说明。
- 中英文长期文档。
- 恢复开发所需的项目自有技能、补丁、资源和工具。

换机恢复的最低验收标准是：从远端全新克隆后，不借用旧工作区文件即可安装依赖、完成构建、运行规定测试并继续修改源码。

生产秘密不通过源码仓库恢复。以下内容必须在 GitHub 外另做加密备份：

- 生产 API 密钥和客户密钥。
- 套装私钥、证书、签发状态和真实授权材料。
- 已使用/未使用套装 ID 的真实分配台账。
- 客户升级所需的套装身份映射。

这类加密备份至少保留两份，并与仓库访问凭据分开保存。GitHub 中只保留生成工具、格式定义和无秘密值模板。

## 双语同步规则

`github.md` 是中文阅读版，`github.en.md` 是英文执行参考版，两份文档必须表达同一套规则。

项目内长期有效的 `.md` 文档也按同一原则维护：

- `xxx.md`：中文阅读版。
- `xxx.en.md`：英文执行参考版。
- 修改长期规则、产品设计、架构说明、Signal 复盘或发布流程时，同时更新两份。
- 临时排查日志沉淀为长期文档时，补齐英文版。
- 第三方依赖、运行缓存和导出产物中的 `.md` 不参与本规则。

## 工作区隔离

GitHub 通版工作必须与业务开发现场物理隔离：

- 业务源码目录是功能来源，不能直接白标化后推送。
- 通版只在 `github-export-baishihappy-2` 隔离目录中生成、检查、提交和推送。
- 不得在业务目录内替换品牌、logo 或数据目录名。
- 不得把通版生成结果反向覆盖业务目录。
- 每次同步必须从一个明确的业务版 Git 提交开始。
- 通版同步提交必须在提交信息中记录 `Source-Checkpoint: <业务版提交哈希>`。

当前业务恢复点为：

```text
485c55e test: rebuild Signal runtime before control checks
```

后续同步时使用新的实际业务提交哈希，不能永远沿用以上旧值。

## 永不进入 GitHub 的内容

禁止提交文件本体、内容片段、日志输出或历史版本中的以下数据：

- API key、Token、Cookie、口令和真实环境变量。
- `license.dat`、授权码文本和客户授权材料。
- 私钥、证书私钥、DPAPI 数据、缓存密钥和机器绑定秘密。
- `.package-secrets`、真实签发器状态和真实套装 ID 台账。
- `taozhuang` 客户套装目录、安装包、压缩交付包和发布产物。
- Signal / WhatsApp / Telegram 登录态、账号数据和实例目录。
- `TranslationCache`、`SignalInstances`、`Partitions`、WebView 用户数据和本地缓存。
- 客户聊天记录、导出的企业资产、联系人资料、图片、视频和语音。
- 客户配置、机器码、用户名和设备绑定记录。
- 本机身份信息、私人绝对路径和可识别客户的信息。
- 开发或客户运行日志。
- `node_modules`、`dist`、`dist-electron`、`.runtime`、`.tmp`。

根目录 BUG 截图、友商截图和临时反馈图片默认不提交。只有用户明确指定作为长期证据归档时，才提交指定文件。

## 每次推送前的固定流程

只有用户明确说“推 GitHub”“上传通版”或“强推通版”时才执行远端推送。普通存档、Git 安全检查或本地提交不自动触发推送。

### 1. 确认来源和隔离目录

- 确认业务工作区提交哈希。
- 确认当前目录是 `github-export-baishihappy-2`。
- 确认没有修改、移动或覆盖业务工作区。
- 确认本次同步覆盖业务提交以来的所有功能、测试和长期文档变化。

### 2. 确认远端、分支和可见性

检查：

```text
git remote -v
git branch --show-current
```

- 远端必须是预期仓库，分支必须是预期分支。
- 每次推送前必须检查仓库可见性并向用户报告。
- 因仓库包含完整授权和安全实现源码，默认建议使用私有仓库。
- 如果仓库为公开状态且用户本次没有明确确认公开推送，停止推送并报告。

### 3. 检查功能同步

- 通版不能遗漏业务版源码、测试、脚本、技能或长期文档。
- 品牌替换不能破坏路径、窗口识别、缓存隔离、授权或打包流程。
- 通版与业务版的实际功能差异必须为零；品牌和无秘密值配置模板差异除外。

### 4. 检查品牌

扫描当前文件名和文件内容，确认不存在：

- “[业务品牌名称]”及其他业务版企业名称。
- `[BUSINESS_BRAND_NAME]`。
- business brand logo、ico、主题图和品牌图片。
- 企业专属路径名、产品名和文档表述。

通版品牌统一使用：

```text
maoyi
```

品牌资源必须使用通版资源替换，不能因直接删除造成空图、布局错位或构建失败。

### 5. 检查敏感文件和内容

- 按“永不进入 GitHub 的内容”逐项扫描文件名、文件内容、暂存区和未跟踪文件。
- 检查 `.gitignore` 是否覆盖运行数据、秘密目录、客户产物和安装包。
- 扫描疑似 API 密钥、私钥头、授权码、机器码和客户路径。
- 如果发现真实秘密，立即停止推送；仅删除工作区文件并不等于已从 Git 历史删除。

### 6. 检查 Git 历史

- 推送前检查待推送提交范围，而不只检查当前文件快照。
- 确认历史中没有秘密、客户数据、安装包、运行缓存或企业品牌残留。
- 若敏感内容曾进入历史，先完成历史清理和复核，再允许推送。

### 7. 构建和实际测试

在通版隔离目录至少执行：

```text
npm.cmd ci
npm.cmd run build
npm.cmd run test:security
npm.cmd run test:signal-render-cache
npm.cmd run test:electron-smoke
```

- 若项目新增了安全或关键运行测试，也必须执行。
- `test:electron-smoke` 产生的临时截图在人工检查后删除，不提交。
- 只有构建通过时，只能报告“构建通过”。
- 只有实际打开程序并检查对应路径后，才能报告 UI 或运行行为已验证。
- 测试失败时不得推送，除非用户在知道具体失败项后明确要求继续。

### 8. 检查 Git 状态和提交内容

执行：

```text
git diff --check
git status --short
git diff --stat
git diff --cached --name-status
git log -1 --oneline
```

- 逐项确认暂存文件只属于通版。
- 提交信息记录本次同步范围和 `Source-Checkpoint`。
- 提交后再次确认工作区状态。

### 9. 推送规则

- 默认使用普通推送。
- 不把 `git push --force` 作为常规命令。
- 确实需要改写远端历史时，必须取得用户对本次强推的明确授权，并使用 `git push --force-with-lease`。
- 远端、分支、隔离目录、品牌、敏感扫描或测试任一项不合格，不执行推送。

### 10. 推送后灾难恢复复核

推送后必须：

- 确认远端分支提交哈希与本地通版提交一致。
- 在新的临时目录从远端全新克隆。
- 不引用业务工作区文件，重新安装依赖并执行构建和规定测试。
- 记录成功项、失败项和未实际验证项。
- 删除临时克隆和测试产物时，不得影响业务工作区或通版隔离目录。

只有远端提交存在且全新克隆复核完成，才能把本次操作称为“已完成异机恢复备份”。

## Git 安全检查、恢复点和远端备份的区别

- **Git 安全检查**：检查路径、差异、品牌和敏感内容，不会产生提交。
- **本地 Git 恢复点**：创建本地提交，可在本机回退，但本机丢失时也会一起丢失。
- **GitHub 远端备份**：通过隔离、检查、提交、推送和全新克隆复核后，才具备异机恢复价值。

三者不是同一个动作，报告时必须分别说明是否完成。

## 固定执行口径

以后用户要求推送 GitHub 时，固定顺序为：

```text
确认业务提交 -> 同步到隔离通版 -> 功能/品牌/敏感信息检查 -> 构建与实际测试 -> 通版提交 -> 推送 -> 全新克隆复核
```

不得跳过隔离目录，不得从业务目录直接推送，不得把本地提交说成已经完成 GitHub 防丢失备份。
