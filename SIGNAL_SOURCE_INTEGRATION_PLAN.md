## 1. 背景与目标

当前项目使用内置 Signal Desktop，并通过 `df-bootstrap.cjs`、运行时 ASAR 补丁和本地控制通道实现启动守卫、多开、窗口贴合、翻译探针和通知桥接。该方案已经能够运行，但依赖对构建产物和压缩代码的字符串补丁，升级、定位和回归成本较高。

下一阶段改为受控的源码级集成，目标是：

1. 保留 Signal 官方的账号数据库、端到端加密、网络协议和消息发送实现。
2. 将窗口控制、消息事件、翻译显示、输入框守卫和通知事件放到稳定的源码边界。
3. 删除宽泛的 `script.execute` 能力，改成固定、可校验、可审计的类型化命令和事件。
4. 降低 DOM 全量轮询、重复缓存查询、重复 DeepSeek 请求以及窗口切换时的闪烁和 CPU 峰值。
5. 保持当前已验收版本可随时回退，源码实验不得直接覆盖客户运行时和登录数据。

### 1.1 `003` 功能兼容合同

- `003` 已按用户决定退出升级交付，但它已经实现的合理 Signal 行为仍是 `006` 源码改造的强制兼容基线。源码改造是升级、提速和加固，不是删功能后重做。
- 内置改造版 Signal 有且只有通过maoyi启动器才能合法启动；禁止直接双击启动，禁止回退或旁路到官方 Signal。
- 原机缺少合法启动凭证时，只拦截退出，不删除原机登录态或数据。
- Signal 整个运行文件夹及其数据被复制到非本机后单独启动时，必须在 Signal 账号数据库和聊天窗口加载前识别外来副本，执行既定的完整应用数据根初始化并退出；初始化失败时保持禁止进入并在下次重试。
- 保留现有多开数据隔离、启动凭证、控制通道、显示/隐藏/位置/关闭/心跳、英文气泡回译中文、中文输入翻译、加密缓存恢复、A/B/A 缓存贴回、未读/通知桥接和现有工作区行为。尚未完成人工验收的能力仍必须保留，不能以“未验收”为由删除。
- 新增的类型化桥、窗口控制器、Windows 所有者窗口关系和主进程锁屏显示门禁只能加强现有行为，不得削弱启动守卫、外来克隆初始化、翻译、缓存、多开或窗口控制。
- 每批源码补丁必须先通过机器可读清单 `signal-source/compatibility/v8.17.0.json` 和现有 Signal 安全/功能回归，再允许进入下一批。任何兼容冲突立即停止，不得用客户 profile 或当前 `.runtime/signal-desktop` 冒险验证。

## 2. 明确不做的事项

- 不把译文写入 Signal 官方消息数据库。
- 不读取或导出 Signal 私钥、会话密钥、附件密钥或账号认证材料。
- 不在本阶段修改 WhatsApp、Telegram 的翻译运行时。
- 不在用户未明确要求时生成安装包、消耗套装 ID 或推送 GitHub。
- 不直接把 `reference-sources/` 中的官方源码归档提交进业务 Git。

## 5. 类型化通信模型

`v8.18.0` 阶段一、二当前已落的线上信封为：

```text
version
sequence
payload.appId
payload.at
payload.type
payload.<按类型固定的字段>
mac
```

其中 `appId` 来自已认证启动会话，并映射回当前 Signal profile（多开实例）；`mac` 使用 HMAC（密钥散列消息认证码）对信封提供完整性校验。`requestId` 只出现在需要请求关联的类型内，不是每个事件都伪造一个请求 ID。阶段二本次新增并已落代码的业务类型是：

```text
主程序 -> Signal
message.snapshot.request
translation.cacheResultBatch

Signal -> 主程序
conversation.changed
message.visibleBatch
translation.cacheResultApplied
```

最终交付的逻辑上下文必须直接携带或由认证会话不可歧义地绑定以下信息；这是最终目标合同，不等同于声称当前线上信封已经逐字段使用该形状：

```text
protocolVersion
suiteId
profileId
requestId（仅请求关联类型）
type
payload
timestamp
```

最终目标中主程序到 Signal 的业务命令白名单：

```text
window.show
window.hide
window.focus
window.setBounds
window.shutdown
theme.apply
message.snapshot.request
translation.cacheResultBatch
translation.composerResult
```

最终目标中 Signal 到主程序的业务事件白名单：

```text
runtime.ready
runtime.heartbeat
window.stateChanged
conversation.changed
message.visibleBatch
translation.cacheResultApplied
message.added
composer.changed
composer.translateRequested
composer.sent
notification.changed
runtime.error
```

消息和缓存标识优先级：

1. `profileId + conversationId + messageId`
2. `profileId + conversationId + sourceHash`
3. 仅在 Signal 无法提供稳定联系人标识时，才使用 `profileId + sourceHash` 兜底。

自定义多开名称不参与缓存主键，因此改名不得造成缓存失效或串号。当前信封必须校验 `version`、单调 `sequence`、HMAC（密钥散列消息认证码）、认证 `appId`、类型、字段长度和请求时效；最终逻辑上下文还必须验证套装与 profile（多开实例）绑定。未知命令直接拒绝。源码桥稳定后删除过渡命令 `script.execute`。

## 6. 状态流程

### 6.1 合法启动

1. 主程序先验证 `license.dat`、本机码和套装绑定。
2. 主程序为指定 `profileId` 生成 256 位随机一次性启动凭证。
3. 凭证有效期保持 10 秒，只能使用一次，并绑定 `suiteId + profileId + Signal processId`。
4. Signal 在账号数据库和窗口初始化前通过 Windows 命名管道验证凭证。
5. 通过后建立类型化运行通道并发送 `runtime.ready`；失败则退出，不加载账号数据。

### 6.2 窗口贴合

1. 主程序是工作区位置和可见性的唯一权威来源。
2. Signal 主进程保存主窗口对象并响应 `show/hide/focus/setBounds/shutdown`。
3. Windows 下实验 `GWLP_HWNDPARENT` 所有者窗口关系，使 Signal 跟随主程序层级、最小化和恢复；禁止跨进程 `SetParent` 嵌入。
4. 所有者关系不可靠时回退到当前独立窗口贴合逻辑。
5. 保持 bounds 同步防抖 `80ms`、恢复/切换 burst `100ms`，不恢复持续 `800ms` 稳态轮询。
6. 拖动期间只做节流后的 bounds 同步，不反复 show/focus/moveTop；拖动结束后做一次最终校准。

### 6.3 已缓存译文显示

1. 会话切换或时间线可见项变化时，Signal 按稳定消息 ID 批量发送 `message.visibleBatch`。
2. 主程序先查询内存热缓存，再批量查询机器绑定加密磁盘缓存。
3. 命中结果通过一次 `translation.cacheResultBatch` 返回。
4. Signal React 组件在气泡内固定译文槽批量渲染，不通过全页 DOM 扫描逐条插入。
5. 译文只作为内存派生视图存在，不写入 Signal 消息数据库。锁定、隐藏、断连、重载或关闭会清除 Signal 渲染侧派生 LRU（最近最少使用缓存）、正文和请求关联，但不会删除机器绑定加密磁盘缓存。

### 6.4 缓存未命中和新消息

1. `message.added`、首次登录后用户主动翻到的历史消息和手动刷新可以进入实时翻译队列。
2. 屏幕上方两屏和下方一屏的预取只用于批量查缓存，不自动消耗 DeepSeek token。
3. 新账号主动翻历史时，缓存未命中的可见历史进入 `history-backfill`；一次收集后批量并发，避免逐条视觉断续。
4. 翻译完成必须先写机器绑定加密磁盘缓存，再更新内存热缓存和当前译文槽。
5. 企业资产不得触发额外翻译请求；人工导出只读取已有原文、译文和附件。

### 6.5 中文输入与发送

1. Signal 源码输入组件直接读取当前会话和当前输入文本，不依赖定时 DOM 探针判断发送。
2. Enter 或发送按钮触发时，只要文本仍含中文，就同步拦截官方发送动作并生成唯一 `requestId`。
3. 翻译结果必须同时匹配 `profileId + conversationId + requestId + sourceHash` 才能写回。
4. 翻译期间重复 Enter/点击只关联同一请求，不重复翻译、不发送中文、不叠加结果。
5. 英文成功写入并校验当前会话未变化后进入“英文待发送”；下一次 Enter/发送按钮才放行官方发送。
6. 用户切到其他联系人时，旧请求可以完成并缓存，但不得写入新联系人的输入框。
7. 发送成功后由源码事件输出 `composer.sent`，状态“已发送”显示 2 秒。

### 6.6 通知和未读

1. 新消息事件携带 `profileId + conversationId + messageId`，不从历史滚动和翻译渲染推导未读。
2. 顶部多开未读数只接受 Signal 官方会话状态的增减事件。
3. 历史滚动、缓存贴回、手动刷新和未读预览不得增加未读数。
4. 通知点击暂不恢复系统级可点击行为，直到实例 token 与消息 ID 的精确绑定完成并通过真实通知验收。

## 7. 固定参数

| 项目 | 参数 |
| --- | --- |
| bounds 同步防抖 | `80ms` |
| 恢复/切换 burst | `100ms` |
| 持续稳态窗口轮询 | 禁用，不使用 `800ms` 循环 |
| 初始缓存预加载 | 最近 `1000` 条 |
| 会话历史缓存分页 | 每页 `1000` 条 |
| 历史译文渲染批次 | 每批 `100` 个气泡 |
| 浏览器/Signal 渲染热缓存 | `1500` 条或 `15MB`，先到者生效 |
| 全局内存热缓存 | `120000` 条或 `320MB`，先到者生效 |
| 全局实时翻译并发 | `24` |
| 普通单 profile 并发 | `1` |
| 交互翻译单 profile 并发 | `2` |
| 首次历史回填单 profile 并发 | 最多 `18` |
| 翻译失败重试 | `5s`、`30s`，最多两次自动重试 |
| 一次性启动凭证 | `256 bit`、`10s`、单次使用 |

以上参数是源码集成首版基线。只有真实 CPU、内存和视觉验收提供证据后才调整，不能因为重构自行改回 `220ms` 或持续轮询。

## 9. 安全边界

- 保留并复用当前外来克隆保护、DPAPI 设备绑定、套装绑定和一次性凭证。
- 启动凭证不得出现在命令行、普通日志、localStorage 或 Signal 数据库中。
- 类型桥拒绝任意 JavaScript、任意文件路径、任意 shell 命令和未知消息类型。
- 翻译缓存继续使用每记录 `AES-256-GCM`，密钥由 Windows DPAPI 保护并绑定 `suiteId + dataRootId`。
- 生产包不保存诊断日志；开发/GitHub 环境可以保留不含消息正文、密钥和账号材料的结构化诊断。
- Signal 官方更新继续禁用，版本升级只能经过源码差异复核、构建和真实验收。
