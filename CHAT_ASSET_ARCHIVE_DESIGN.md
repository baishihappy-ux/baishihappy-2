# 企业聊天资产归档设计

## 定位

聊天记录不是临时缓存，而是企业资产。

maoyi需要把员工在 Signal、WhatsApp、Telegram 私聊中产生的沟通内容、翻译结果、成交经验、失败案例、典型错误样本统一沉淀下来。随着客户端使用人数增加，这些数据会形成企业软实力：新员工上岗后可以通过 AI 建议更快理解客户意图、复用优秀话术、规避常见错误，并提高成交率。

## 核心原则

1. 多开关闭不等于资产删除。
   - 用户关闭某个多开时，运行资源必须释放。
   - 已采集的聊天资产必须继续保存。

2. 仍然打开的多开必须保持接收通知。
   - 为了省资源而自动关闭 Signal 多开，会导致对应账号无法实时收通知。
   - 资源优化不能破坏“打开即在线、打开即收通知”的业务要求。

3. 不做群聊模型。
   - 当前业务只考虑一对一私聊。
   - 数据结构不保留群名、群成员、群消息线程等字段。

4. 原始聊天数据与 AI 训练数据分层。
   - 原始聊天记录用于审计、检索、复盘。
   - AI 使用前需要清洗、脱敏、标注和结构化。

## 进程与会话生命周期

Signal 是独立桌面客户端实例。

- 每个 Signal 多开使用独立 `signalDataDir`（Signal 独立数据目录）。
- 用户关闭某个 Signal 多开时，必须按该目录精准结束对应 `Signal.exe`（Signal 进程）。
- 用户保留的 Signal 多开必须继续运行，以保证通知接收。

WhatsApp 和 Telegram 以网页会话为主。

- 用户关闭多开时，需要销毁对应 `webview`（内嵌网页视图）或会话承载对象。
- 关闭后不应保留后台 WebSocket、定时器、翻译任务或不可见页面资源。
- 关闭运行资源不等于删除已经归档的聊天资产。

## 统一数据模型

建议使用“企业 -> 操作员 -> 平台账号/多开 -> 私聊联系人 -> 消息”的链路。

### 私聊联系人

- `contact_id`：系统内部联系人 ID。
- `enterprise_id`：企业 ID。
- `operator_id`：操作员 ID。
- `platform`：Signal、WhatsApp、Telegram。
- `profile_id`：多开实例 ID。
- `profile_name`：多开实例名称，例如 SG01、WS01、TG01。
- `external_id`：平台侧唯一标识，例如手机号、username、Signal 标识。
- `phone_number`：手机号，可为空。
- `username`：用户名，可为空。
- `display_name`：平台昵称。
- `remark_name`：备注名。
- `avatar_asset_id`：头像资源 ID，可为空。
- `last_message_at`：最近一次会话时间。
- `created_at`：首次归档时间。
- `updated_at`：最近更新时间。

### 消息

- `message_id`：系统内部消息 ID。
- `platform_message_id`：平台侧消息 ID，可为空。
- `enterprise_id`：企业 ID。
- `operator_id`：操作员 ID。
- `profile_id`：多开实例 ID。
- `contact_id`：私聊联系人 ID。
- `direction`：`inbound`（对方发送）或 `outbound`（我方发送）。
- `message_type`：`text`、`image`、`video`、`file`、`voice`。
- `source_text`：原文。
- `translated_text`：译文。
- `source_language`：原语言。
- `target_language`：目标语言。
- `sent_at`：平台发送时间。
- `received_at`：客户端接收时间。
- `archived_at`：归档时间。
- `quoted_message_id`：引用的消息 ID，可为空。
- `send_status`：发送成功、发送失败、未知。
- `is_deleted`：是否撤回或删除。
- `raw_payload_ref`：原始采集片段引用，可用于排查。

### 附件

图片、视频、文件、语音不直接塞进消息表。

- `asset_id`：附件资源 ID。
- `message_id`：关联消息 ID。
- `asset_type`：图片、视频、文件、语音、缩略图。
- `storage_path`：本地或服务端对象存储路径。
- `sha256`：文件哈希，用于去重和防篡改。
- `mime_type`：媒体类型。
- `file_name`：文件名。
- `file_size`：文件大小。
- `duration_ms`：音视频时长，可为空。
- `width`、`height`：图片或视频尺寸，可为空。
- `ocr_text`：图片 OCR 识别文本，可为空。
- `created_at`：归档时间。

### AI 标注

- `label_id`：标注 ID。
- `message_id` 或 `conversation_id`：关联消息或会话。
- `label_type`：成功案例、失败案例、典型错误、优秀回复、敏感风险、客户拒绝、成交信号。
- `label_source`：人工标注、规则标注、AI 预标注。
- `confidence`：置信度。
- `note`：人工备注。
- `created_by`：标注人。
- `created_at`：标注时间。

## 存储格式

本地客户端：

- 使用 `SQLite`（本地嵌入式数据库）保存结构化聊天事件。
- 附件保存为文件，数据库只记录路径、哈希、类型和关联消息。
- 本地先落库，避免断网丢数据。

企业服务端：

- 使用 `PostgreSQL`（结构化数据库）保存联系人、消息、标注、同步状态。
- 使用对象存储保存图片、视频、文件、语音等附件。
- 客户端断网时排队，联网后补传。

AI 数据导出：

- 使用 `JSONL`（一行一条 JSON 记录）作为训练或评估数据格式。
- 使用 `CSV`（表格格式）给运营和管理人员做人工筛选。
- 训练前必须脱敏手机号、姓名、邮箱、地址、订单号、付款信息等敏感内容。

## 采集策略

WhatsApp / Telegram：

- 通过网页注入脚本读取可见消息、新消息、发送事件、联系人信息和翻译结果。
- 需要做去重，不能因为页面重绘重复写入同一条消息。

Signal：

- 通过独立 Signal 窗口运行时采集运行期间可见的消息、通知事件、发送事件和翻译事件。
- 不假定可以一次性完整读取所有历史消息。
- 完整历史采集需要单独做 Signal 适配研究。

翻译器自身：

- 所有翻译请求和翻译结果必须归档。
- 原文、译文、语种、使用模型、操作员、多开实例、联系人必须能关联。

## AI 使用路线

第一阶段：统一归档。

- 先把聊天记录、翻译结果、附件、联系人统一保存。
- 保证可检索、可追溯、可同步。

第二阶段：复盘和标注。

- 管理员可以把会话标记为成功案例、失败案例、典型错误、优秀回复。
- 标注质量决定后续 AI 建议质量。

第三阶段：AI 建议。

- 基于当前会话上下文，检索历史相似案例。
- 优先使用 `RAG`（检索增强生成：从历史资料中找相似案例再生成建议）给员工推荐回复。
- 在数据足够干净、标注足够稳定后，再评估模型微调。

第四阶段：培训新员工。

- 新员工打开客户会话时，系统给出历史相似成交案例、常见误区和建议回复。
- 员工发送前可对建议话术进行人工确认和修改。
- 管理端持续沉淀优秀回复和错误样本，形成企业话术资产。

## 后续实施顺序

1. 先定义 SQLite 本地表结构。
2. 接入 WhatsApp / Telegram 的可见消息采集。
3. 接入翻译器自身的翻译记录归档。
4. 接入 Signal 运行期间消息和事件采集。
5. 做附件归档和去重。
6. 做企业服务端同步。
7. 做管理端检索、复盘、标注。
8. 做 AI 建议和新员工辅助回复。

## 联系人归档与排除规则（2026-07-10）

- 我方中文输入不属于企业聊天资产，不写入翻译缓存或企业资产。
- 若对方历史首次消息或最新消息为中文，则将该联系人持久标记为无需缓存、无需归档。
- 判定只使用对方发送的正文；我方消息和引用区中文不触发标记。
- 标记后删除该联系人的既有翻译缓存和企业聊天档案，并拒绝后续写入。
- 企业资产总入口为 `EnterpriseChatAssets/index.html`，可按联系人、平台或多开搜索。
- 每个联系人拥有独立 `index.html`，按日期和收发方向显示英文原文、中文译文与时间。
