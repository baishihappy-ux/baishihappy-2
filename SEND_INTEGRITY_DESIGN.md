# 聊天发送完整性设计

## 安全目标

防止员工屏幕显示的收件人、正文、收款账号、金额或钱包地址与平台实际发送内容不一致。企业资产防篡改不是本阶段重点，重点是发送中的内容完整性和可信确认。

## 普通消息发送事务

1. 翻译完成时保存预期英文正文和当前 `profileId + contactId`。
2. 第二次回车前重新读取输入框和当前联系人。
3. 平台发送事件触发前再次逐字符读取正文。
4. 三次正文和联系人必须完全一致，否则阻止发送。
5. 发送后读取新产生的我方气泡，与预期正文逐字符比较。
6. 不一致时停止该实例继续发送并显示明确告警。
7. 我方中文输入只在输入框和翻译内存中短暂存在，不写缓存、企业资产或运行日志。
8. 翻译内容不通过剪贴板中转。

## 单一数字收款账号

- 当消息仅包含数字及允许的空格，并满足敏感数字长度规则时，不调用翻译接口。
- 固定长度规则为去除内部半角空格后 `6–34` 位数字；首尾空格、换行和其他字符不允许。
- 全程使用字符串，禁止转换为 JavaScript `Number`，避免前导零和大整数精度丢失。
- 第一次回车锁定账号、完整原文和收件人，不发送。
- 主程序可信区域显示收件人和完整账号。
- 第二次回车重新核对后发送。
- 发送后再次核对我方气泡。
- 禁止自动增删空格、格式化、剪贴板中转和 AI 改写。

## 单一 USDT 钱包地址

- 不调用翻译接口，不改变大小写和任何字符。
- 使用成熟区块链地址库校验，不只使用正则表达式。
- 支持识别和校验 TRC20、ERC20、BEP20、Solana 等地址格式。
- ERC20 与 BEP20 地址外观可能相同，网络必须由人类确认。
- 第一次回车锁定地址、网络和收件人。
- 主程序可信区域显示网络、完整地址、校验结果和收件人。
- 第二次回车重新核对后发送。
- 地址校验失败或网络不明确时禁止发送。
- 发送后再次逐字符核对我方气泡。
- TRC20 与 Solana 使用审计过的 Base58/Base58Check 库校验；EVM 地址使用 Keccak-256 校验混合大小写校验和。
- `0x` 地址格式不能区分 ERC20 与 BEP20，第一次确认时必须人工选择网络。

## 确认状态机参数

- 第一次回车由三平台页面守卫拦截，不触发平台发送，也不调用翻译接口。
- 主进程原生确认框显示平台、多开、联系人、完整账号或地址和网络；Signal 确认时临时隐藏其独立窗口，确认结束后恢复贴合。
- 确认成功后生成 32 字节随机凭证，只保存在主进程内存中，有效期 `120 秒`，只能使用一次。
- 第二次回车重新核对 `profileId + platform + 联系人签名 + 完整文本 + 网络`，任一变化即作废。
- 主程序顶部多开区域显示可信确认条；它不属于 WhatsApp、Telegram 或 Signal 页面 DOM。
- 发送后在累计约 `3 秒` 的观察窗口内读取新产生的我方气泡并逐字符核对。

## 可信显示区域

支付类敏感信息不能只显示在 WhatsApp、Telegram 或 Signal 页面中。主程序自身需要提供不属于平台网页 DOM 的可信确认区域，显示：

- 平台及多开
- 联系人
- 金额和币种（如有）
- 收款账号或钱包地址
- 钱包网络（如有）
- 地址校验结果
- 确认倒计时或二次回车状态

## 客户端安全边界

- WhatsApp 与 Telegram 远程页面不能调用主程序 IPC；Signal 脚本执行入口只对受 ASAR 完整性保护的本地主渲染页开放。将 Signal 动态脚本迁移为完全类型化命令仍属于后续纵深加固项。
- WebView 只允许 WhatsApp 和 Telegram 官方域名。
- 打包版关闭开发者工具、远程调试、诊断输出和运行日志。
- 启用 ASAR 完整性、Electron Fuses 和关键文件哈希检查。
- 开发版只使用项目受控 Signal；打包版只使用安装包内置 Signal；禁止官方 Signal 回退。
- 管理员级恶意程序仍可能截屏、键盘记录、读取内存或替换整个程序，这是本地客户端无法绝对消除的系统边界。

## 2026-07-12 验证范围

- 构建、类型检查、地址校验单测、一次性凭证源码门禁和发送后气泡核对代码路径已通过自动检查。
- Electron 客户端与授权程序已在隔离数据目录实际打开并完成窗口截图冒烟测试。
- 真实 WhatsApp、Telegram、Signal 的“第一次回车确认、第二次回车发送、发送后气泡核对”尚未由人类实际验收，因此不能宣称三平台 UI 已验收。
