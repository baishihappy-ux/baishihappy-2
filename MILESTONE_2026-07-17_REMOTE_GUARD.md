# 2026-07-17 御前侍卫阶段恢复点

## 阶段结论

“御前侍卫”已经进入当前源码，形成一层 Windows 用户态远控发现与快速止损能力。它覆盖高置信的活动 RDP 会话和标准 VNC 入站连接；发现后先锁定maoyi工作区可见性，再发出多显示器全屏警报和声音，3 秒后请求 Windows 锁屏。

这不是杀毒软件，也不是完整的终端检测与响应系统。未知远控协议、自定义 WebSocket/HTTPS 通信、进程注入、改名程序、同权限恶意程序终止主程序、管理员或内核权限攻击均可能绕过当前用户态守卫。

## 已落代码

### 1. 轻量巡检

- 快速巡检周期：8 秒。
- 快速巡检内容：活动 Windows RDP 会话、TCP 已建立连接中的本机 VNC 服务端口 5800 至 5999。
- 快速系统命令超时：900 毫秒。
- 已知远控进程补充巡检：5 分钟一次，进程命令超时 1.2 秒。
- 已知进程覆盖 AnyDesk、TeamViewer、RustDesk、ToDesk、向日葵、Windows 快速助手、Windows 远程协助、Chrome 远程桌面、ScreenConnect、Splashtop、Supremo、DWService、MeshCentral、Parsec 和常见 VNC 服务。
- 仅发现已知远控进程时显示黄色风险提示，不直接锁定 Windows，避免把“安装但未连接”误判为正在入侵。
- 巡检失败时显示“巡检受限”，不进行高频重试。

### 2. 高置信远控响应

- 发现活动 RDP 或 VNC 入站连接时，主进程立即锁定工作区显示门禁。
- Signal、WhatsApp、Telegram 内容立即遮罩；Signal 隐藏未确认时沿用失败关闭保护。
- 每个显示器创建独立的置顶全屏警报窗口。
- 系统警报声每 850 毫秒循环一次。
- 发现后 3 秒调用 Windows `LockWorkStation` 锁定工作站。
- Windows 解锁或系统恢复后立即复查；远控仍存在时重新进入 3 秒锁屏响应。
- 远控迹象消失后仍保持事件锁定。已设置客户端锁屏密码时必须验证密码，未设置时必须在警报界面确认，才能恢复工作区。
- 远控仍存在时不能通过普通关闭按钮或客户端密码释放工作区。

### 3. 御前侍卫界面

- 工作区左侧栏新增“御前侍卫”入口，位于锁屏与展开按钮之间。
- 状态包括启动中、值守中、发现可疑程序、巡检受限、发现远控和等待确认。
- 状态面板展示最近巡检时间、本次用时、Windows 会话覆盖情况、远控进程情况和发现项。
- 巡检周期文案定稿为“每 8 秒巡查一次潜在远程连接”，秒数继续读取实际运行参数。
- “远控进程”正常状态文案改为“未发现”。
- 图标采用 `御前侍卫高清.png` 重新抠出的透明人物轮廓，无外圈，显示比例为原基准的 0.8，保留绿色、黄色、红色状态灯并适配黑金与粉色主题。
- 御前侍卫图标和文案下移 2px；展开按钮与 WhatsApp、Telegram、Signal 三个平台入口同步下移 2px，同时保持原有选中和未读动画相对位移。
- 在 Signal 多开界面打开御前侍卫面板时，当前 Signal 原生窗口暂时隐藏；关闭面板后恢复原多开，不刷新、不重启、不切回 SG01。WhatsApp 和 Telegram 也进入相同的阻塞弹窗门禁。

### 4. 通信与打包边界

- 主进程维护检测、事件锁定、警报窗口和 Windows 锁屏状态。
- 渲染层只通过受限 IPC 获取状态、发起人工巡检和请求事件确认。
- 普通客户包和套装客户包的文件清单均显式加入 `remote-guard.js` 与 `remote-guard-core.js`。
- 安全门禁会阻止缺少御前侍卫模块的客户包继续构建。
- 御前侍卫测试已加入完整安全回归命令。

## 性能与误报取舍

- 快速巡检仅运行短时 Windows 系统命令，不抓包、不录屏、不扫描磁盘、不使用高频 WMI 或 PowerShell。
- 已知远控进程使用 5 分钟缓存补查，降低 CPU 和进程枚举开销。
- 只有活动 RDP 和标准 VNC 入站连接被视为高置信远控；进程存在本身不自动锁屏。
- 检测命令不可用时进入“巡检受限”，避免因系统命令偶发失败造成无限锁屏循环。
